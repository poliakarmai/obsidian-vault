# AI Loops — заметки для оркестрации

Источник: пользователь, 22.06.2026

## 1. Что такое AI Loop

Замкнутый цикл: агент действует → проверяет результат → корректирует → повторяет.

**Три обязательных компонента:**
- **Проверка (verification)** — автоматическая валидация результата. Без неё loop = сжигание токенов.
- **Состояние (state)** — что уже сделано, что нет. Без состояния loop теряет контекст.
- **Лимиты (bounds)** — макс. шагов, бюджет, таймаут. Без лимитов = бесконечный цикл.

**Где работает:** кодогенерация, рефакторинг, тестирование, документооборот.
**Где не работает:** творческие задачи без чётких критериев успеха.

## 2. Loop Engineering — практика

**Инструменты:**
- `STATUS.md` — единый файл состояния (что сделано / что нет / что сломалось)
- `skills` — процедурная память агента (не изобретать заново каждый раз)
- `worktrees` — изолированные git-ветки для параллельной работы
- `MCP` — инструменты агента (файлы, терминал, API)
- `субагенты` — параллельные работники с чёткими задачами
- `бюджетные стопы` — авто-остановка при превышении лимита токенов/времени
- `утренний отчёт` — вместо стены логов — одна сводка утром

**Принцип:** агент кодит ночью → утром разработчик читает отчёт, а не логи.

## 3. Один loop, 259 PR/мес

**Схема из 6 частей:**
1. Задача (чёткая, атомарная)
2. Действие (агент выполняет)
3. Проверка (авто-тесты, линтер, judge)
4. Ограничитель (max 3 попытки, бюджет токенов)
5. Результат (PR или отказ с объяснением)
6. Отчёт (что сделано, что нет, почему)

**Типичные сбои:**
- Агент зацикливается на одной ошибке → нужно правило трёх провалов
- Агент меняет подход вместо фикса конкретной ошибки → verification loop
- Агент превышает бюджет → авто-стоп

**Дешёвые альтернативы:**
- Вместо полного loop → один запрос с самопроверкой
- Вместо ночного кодинга → фоновый делегат с таймаутом
