## Главная идея

Вертикальный агент — агент под конкретный класс задач (таблицы, финансы, продажи, DevOps). Базовый каркас (модель → инструменты → ответ) делается за день. Сложная часть — система контекста.

> **Формула:** хороший агент = точное сжатие распределения задач в своём домене.

---

## Три уровня контекста (как CPU-кеш)

### L1 — всегда в контексте (~80% задач)
- Маленький, мгновенный, сжатый
- Дорого строить, но окупается на каждом вызове
- Пример: контракт `execute`, базовые типы, частые методы

### L2 — по запросу (~15%)
- Кураторские спецификации, загружаются одним шагом
- Не платишь токенами пока не нужно
- Пример: `getConditionalFormattingInfo()` при работе со сводными таблицами

### L3 — сырой справочник (~5%, длинный хвост)
- Полный, машинно-генерированный, неудобный
- Рядом — навык-карта с grep-командами
- Пример: `grep -n '"charts.add"' api-reference.json`

## Ключевые приёмы

### 1. Один execute вместо десятка мелких инструментов
Модель принимает одно решение — «напиши код». Код вызывает API. Нет шума от выбора инструментов.

### 2. Сжатие L1 — формула-алиасинг
Повторяющиеся формулы → `F1`, стили группируются в диапазоны, авто-контекст из заголовков.

```
=F1 -> =RC[-2]*RC[-1]
D2:D201: 200 ячеек, формат #,##0.00, зелёный текст
```

### 3. Сжатая обратная связь (diff + linter)
После записи — сводка изменений, а не сырой список. Подозрительные случаи (`#REF!`) поднимаются наверх.

---

## Как применить к своему проекту

1. Составить карту задач: L1 (частые), L2 (важные периодические), L3 (хвост)
2. Решения о повышении уровня — по реальным сессиям и ценности результата
3. Частота ≠ единственный сигнал. Сводные таблицы редкие, но impact большой → L2
4. Скучная, но ломающая всё деталь → L1. Красивая, но редко нужная → L3

## Применимость к Hermes

| Уровень | У них | У нас |
|---|---|---|
| L1 | execute + базовые типы | bybit-trading skill, конституция, протокол |
| L2 | getToolInfo() | skill_view() — загрузка по требованию |
| L3 | grep по api-reference.json | web_search, web_extract, session_search |

Наша система скиллов — это ровно L1/L2/L3 иерархия. bybit-trading всегда в контексте (L1), остальные 147 скиллов по требованию (L2), интернет и поиск по сессиям (L3).
